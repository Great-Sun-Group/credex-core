Technical Foundations and Implementation

Information being stored, retrieved, and processed by the credex ecosystem

The most critical pieces of information in the credex ecosystem are stored in smart contracts that behave in trusted and mutually agreed ways within the ground rules of the ecosystem. The ground rules of the system, which are encoded in credloops and credcoin, are the core of this invention. The critical pieces of information retrieved to implement the invention include:

  1. Credexes conceptualized, modelled and implemented as connections between member nodes in the credex ecosystem. Programmatically/visually expressed in its most simple form: (member)-[CREDEX]->(member).
    1. Whether the credex has a current outstanding balance, and the CXX value of that balance.
    2. The denomination of the credex.
    3. The current CXXmultiplier of the credex, by which the CXX value is divided by to provide the face values for credexes denominated in anything other than CXX. The CXXmultiplier is one of the unique aspects of this invention, and is explored in more detail in the Credcoin section below.
  2. Daily exchange rates
    1. Retrieved into the programmatic space of the ecosystem expressed as values in ounces of gold (XAU).
    2. Immediately processed and stored as CXX values.
  3. Formulas
    1. The value of each credex can be updated daily according to custom formulas based on the exchange rates and market rates imported. Credcoin itself is the core implementation of this approach, which enables additional formulas to be overlaid as their own "denominations" (protocols under development).
  4. DCO gift variables set by participating members and their DCO security account holders.
    1. Amount to be given daily in CXX.
    2. Denomination of DCO linked currency account for gift.
    3. Verification that currency account balance is sufficient for DCO participation at member set levels.
  5. DCO market variables set by participating businesses along the value production and consumption chains of the network.
    1. Amount to be paid daily in CXX, which will primarily be determined by market mechanics (see credcoin section).

How is this information acquired, stored, processed, and displayed?

Data acquisition

  1. Credexes
    • Requires digital signatures by two recognized member nodes. This information is acquired via the credex app or api. Credexes are offered or requested by one party, and accepted or declined by the other party. An accepted credex becomes part of the critical information of the invention.
  2. Daily exchange rates
    • Programmatically queried from a third party every 24 hours just after midnight UTC during the DCO.
  3. Formulas
    • Theoretically from any source.
  4. DCO gift variables
    • Set by participating members via the credex app or api.
    • Verified by participants' chosen DCO security account holders.
  5. DCO market variables (implementation under development)
    • Set by participating members and their businesses via the credex app or api.
    • Prices and values set by markets to be established within the ecosystem.

Data storage and processing

Ecosystem data is currently stored in Neo4j databases on a Google/Neo4j Aura cloud server and/or AWS EC2 instance with automatic backups. Programmatic processes and protocols to access the data and implement the invention are executed with the credex-core codebase. Data storage and processing solutions for the ecosystem will evolve, especially as web3 architecture is built out.

Data output

Data is output in json format in response to API calls.

Physical components required for data acquisition

  1. Member supplied devices (phones, browser, etc) or member owned servers making api calls to the ecosystem.
  2. Api connection to currency rate data.

Steps involved in acquiring and processing the data

1. Initial value of CXX set arbitrarily to $1.00 CAD

This step will happen once, at the initialization of the ecosystem, and is the only arbitrary variable ever required.

2. Ecosystem members create accounts and initial DCO participants set their DCO variables

3. DCO executed daily

Sets exchange rates in the ecosystem for the next 24 hours and re-calculates existing credex values and CXX multipliers at the new exchange rates. See the detailed DCO documentation.

4. Accepted credex

A credex is offered or requested, then accepted.

5. Identify any credloops formed by the accepted credex, and clear a payable vs a receivable for all credlooped members

The addition of each smart contract to the ecosystem may create opportunities to close one or more credloops with existing credexes. These opportunities must be identified, prioritized, and processed. See the detailed MTQ documentation.
